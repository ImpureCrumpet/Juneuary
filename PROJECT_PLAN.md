# Juneuary — Project Plan

A living plan that captures **where the project is**, **why it was built that
way**, **what's worth knowing** before extending it, and **what to do next**.
For day-to-day usage, see [`README.md`](./README.md).

---

## Status (as of this writing)

- **Data model:** multi-city, concept/occurrence split. Two cities loaded
  end-to-end (Seattle, San Francisco). All weather thresholds, timings, and
  climate-sensitivity flags live on the *occurrence*, not the concept.
- **Source of truth:** YAML files in `data/` and `data/cities/`. The SQLite
  DB is regenerated by `scripts/build_db.py` and is never hand-edited.
- **Live data:** `scripts/fetch_weather.py` pulls daily weather and air
  quality from Open-Meteo, routing automatically between the ERA5 archive
  (older days) and `/forecast` (recent week). Observations + classifications
  are persisted to SQLite. Re-runnable safely via `--skip-existing`.
- **Classification:** three-tier (primary / secondary / triggered) in
  `scripts/classify.py`. Used by both the `propose` CLI and the live
  fetcher, so hypothetical and observed days are scored identically.
- **Anomaly + aberration handling:** observations are scored against
  monthly `city_climate_normals`. Far-out one-offs are flagged as
  aberrations — recorded as data, but excluded from "propose new
  microseason" output and downgraded in confidence.
- **Engine package + API:** the runtime-agnostic core lives in
  `src/juneuary/` (presentation, state, predict, locate). A zero-dependency
  stdlib HTTP API (`serve.py`) is the single front door — it can serve a
  stored day, a forecast window, or fetch+classify an arbitrary date range
  (`/v1/days`) so consumers never touch the DB or Open-Meteo directly. The
  response shape is versioned and documented in `contract/`.
- **Reporting (on the `report-builder` branch):** `scripts/report.py` renders
  a markdown year-to-date report as a pure **consumer of the API** — it pulls
  a classified range from `/v1/days` and normals from `/v1/normals` and
  renders through narrative templates. It does not live on `main` (which is
  just db + logic + api).

## Editorial decisions (and why)

These are the judgment calls baked into the data. They're listed here so
they can be challenged without spelunking through the YAML.

| Decision | Why |
| --- | --- |
| **Concept / Occurrence split.** "Spider Season" is one row globally; each city decides whether/when it happens. | Avoids duplicating universal vibes across cities while keeping local timings and thresholds editable per city. |
| **Constants are *secondary*, not primary.** Spider Season, Karl the Fog, Convergence Zones ride alongside the weather-of-the-day classification rather than competing with it. | They're background traits — saying "today is Spider Season" doesn't preclude also saying "today is Third Winter." |
| **Triggered events are their own tier.** Find Bananas, Praise the Sun, Choking Smoke, Photon Fraud fire when a specific signal is present, independent of the calendar. | These are *events*, not seasons; lumping them in with the primary class made every snowy day look like 5 simultaneous microseasons. |
| **First-sun events have strict gates.** Praise the Sun / Glorious Sun require N prior overcast days, a cloud-cover ceiling, a solar-elevation floor, and (for Glorious) a temperature floor. | Without gates, every clear day in March triggered "first sun" — useless. Thresholds live in `classify.py:FIRST_SUN_THRESHOLDS` for now (see next moves). |
| **Aberrations are flagged, not silenced.** Outlier days (≥20°F anomaly, or out-of-season smoke/snow) are stored in observations, but they don't propose new microseasons. | One freak day shouldn't redefine the model, but the historical record needs to be honest about it happening. |
| **Climate normals are monthly, not daily.** Sourced from NOAA NCEI 1991–2020 30-year normals (approximated in YAML for now). | Daily normals are richer but heavier; monthly is enough to compute meaningful anomalies for the use cases here. |
| **Juneuary is a Seattle thing.** SF's analogous June phenomenon is "June Gloom," coded as a separate concept. | The vibes are different (Seattle: cool + rainy; SF: overcast marine layer) — collapsing them would muddy both. |
| **Karl the Fog is a constant**, not a primary classification. | Like Spider Season — it's an ongoing presence, not a weather of the day. |

## Things worth knowing about

Surprises, limitations, and footguns. Read this before changing things.

### Data quality / API quirks

- **Open-Meteo `/forecast` quietly returns nulls past ~28 days back**, even
  though the docs advertise a 92-day `past_days` max. The fetcher works
  around this by using the ERA5 archive for anything older than 7 days
  and the forecast endpoint only for the last week. The seam is hard-coded
  as `ARCHIVE_DAYS_AGO_THRESHOLD` in `scripts/fetch_weather.py`.
- **Archive has a ~5-day delay** (it's ERA5 reanalysis). The 7-day overlap
  with `/forecast` covers this.
- **Air-quality API range is shorter than weather.** AQ data is best-effort;
  any day the AQ endpoint doesn't cover comes through as null PM2.5/AQI.
- **One run = 2 calls per city** (weather + AQ), regardless of date range
  length. The free Open-Meteo tier (~10k calls/day) is hard to come close
  to with this app unless we add per-day calls — don't do that.

### Classification edges

- **Find Bananas and Convergence Zones don't have temperature ranges** —
  they're signal-gated, so they should only fire on the signals (snow in
  forecast, marine convergence). They no longer match every cold day.
- **Glorious Sun has *never* fired in Seattle YTD** in our test runs.
  That's intentional — its bar (35° solar elevation + warm + ≥3 days of
  prior overcast + low cloud cover) really is rare. If you want it to
  fire more, edit `FIRST_SUN_THRESHOLDS`, but the prior calibration was
  matching it on snow days, which was clearly wrong.
- **Persistence requirement is not yet enforced.** "Hell's Front Porch"
  ideally needs 3+ consecutive hot days; right now a single 92°F day
  matches. Tracked under next moves.
- **Series order is over-represented in the report.** Because winter and
  spring-series can both fire on the same calendar day (Seattle's whole
  joke), the timeline shows many short Winter spans interleaved with
  Fool's Spring spans. This is *accurate*; if you want a cleaner story,
  collapse to "longest dominant series per week" in a future report.

### Solar geometry

- We compute **max solar elevation at solar noon** for the observation
  date and city latitude using a closed-form approximation
  (NOAA-style). It's accurate to within a fraction of a degree —
  good enough for "Photon Fraud" gating. Don't use it for astronomical
  applications.

### Aberration model

- Threshold is a flat `±20°F` anomaly against the start-month normal,
  plus out-of-season smoke (winter) or snow (summer). This is
  globally applied. A per-city threshold derived from each city's
  historical temperature standard deviation would be better; tracked
  under next moves.
- Aberrations are recorded in `observations.is_aberration` /
  `aberration_reason` and visible via `query.py aberrations`.

### API + reports

- **`main` is just db + logic + api.** The report generator was branched off
  to `report-builder`, where it consumes the HTTP API instead of the DB.
- `/v1/days` fetches Open-Meteo (archive for past, forecast for future) and
  classifies in-memory — it does **not** persist. The report therefore makes
  live calls through the API at render time (2–4 Open-Meteo calls per city per
  range), rather than reading pre-fetched observations.
- `/v1/state` and `/v1/forecast` still read stored observations
  (`fetch_weather.py --skip-existing` to top up); only `/v1/days` is live.
- The contract is versioned via `SCHEMA_VERSION`; bump it and update
  `contract/state.schema.json` + `contract/example_state.json` when the DTO
  changes. Server fetchers are injectable, so the live routes are testable
  without network.

## Next moves

Ordered loosely by ROI / how blocked other work is on each.

### Near-term (≤1 day each)

- [ ] **Move first-sun + signal-gate thresholds out of `classify.py`** and
      into per-city YAML (e.g. `data/cities/seattle.yaml:classifier`).
      Currently hard-coded; only Seattle is calibrated.
- [ ] **Per-city aberration thresholds**, derived from each city's monthly
      temperature standard deviation. Replaces the flat ±20°F.
- [ ] **Persistence requirement for heat events.** Hell's Front Porch and
      Oppressive Sun should require N consecutive days above threshold,
      not just one.
- [ ] **`scripts/reclassify.py`** — re-run the classifier across all
      observations without re-fetching. Lets us iterate on `classify.py`
      without rate-limiting ourselves.
- [ ] **Cron / scheduled job:** daily `fetch_weather.py --all --days 2
      --skip-existing` to keep all cities up to date with one nightly
      pull per city.

### Medium-term (a few sessions)

- [ ] **Non-West-Coast city** (NYC, Boston, or Chicago) to stress the
      model with hurricanes, polar vortex events, real winter, and
      humid summers. Will surface concept gaps and probably motivate
      a few new microseasons.
- [ ] **Candidate microseasons tracker.** When the same anomaly pattern
      recurs across N years on roughly the same calendar window, flag
      it as a candidate-microseason for human review. Bootstraps new
      concepts from data instead of from folk knowledge.
- [ ] **Span-collapsing report mode.** A weekly "what was the dominant
      microseason this week" view, alongside the daily detail.
- [ ] **Real NOAA NCEI ingestion.** Replace the approximated monthly
      normals with the official 1991–2020 product.

### Longer / optional

- [ ] **JS frontend** (pnpm + a small framework — likely Astro or
      SvelteKit given how static this is) over the **JSON API**, the same
      way the report now consumes it. The engine and contract are already
      deploy-ready; the frontend is just another presentation client.
- [ ] **Prediction surface.** `/v1/days`/`/v1/forecast` already classify
      future days; build a dedicated forecast/prediction client on top.
- [ ] **Hourly observations** instead of daily, for sub-day phenomena
      (Convergence Zone Showers, marine layer push). Adds API cost.
- [ ] **Map view** — animate the season as it moves across cities.
      Pure bonus.
- [ ] **CSV / iCal export** of the timeline.
